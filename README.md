# Rust Toolchain Action

A GitHub Action that installs a Rust toolchain using `rustup` see the [docs](https://rust-lang.github.io/rustup/concepts/index.html), reading
`rust-toolchain.toml` by default with action input overrides. Inspired by
[dtolnay/rust-toolchain](https://github.com/dtolnay/rust-toolchain) with
extended channel resolution, explicit `profile` support, and a fluent builder API.

## Features

- **TOML-first** — reads `rust-toolchain.toml` from the workspace root
- **Override by inputs** — action inputs (`toolchain`, `targets`, `target`, `components`, `profile`) take precedence over toml
- **Expressive channel resolution** — supports `stable N (year|month|week|day) ago` and `stable minus N releases`, resolving to a `<major.minor>` series that rustup pins to its newest patch
- **Flexible component parsing** — accepts comma, space, or newline-separated components
- **Fluent Builder API** — programmatic `ToolchainSpecBuilder` with method chaining
- **Cache-key compatible** — `cachekey` output matches dtolnay/rust-toolchain format for seamless cache interop
- **Profile support** — set `profile` via `rust-toolchain.toml` or input (minimal, default, complete). A requested profile is always passed to rustup explicitly, so a runner-wide `rustup set profile` cannot silently change it
- **Overrides that actually apply** — exports `RUSTUP_TOOLCHAIN`, which outranks `rust-toolchain.toml` in rustup's [override precedence](https://rust-lang.github.io/rustup/overrides.html), so every later step runs the toolchain you asked for

## Quick Start

```yaml
- uses: elioseverojunior/rust-toolchain@v1
```

Without any inputs, the action reads `rust-toolchain.toml` in the workspace root.
If the file doesn't exist, defaults to `stable`.

### With Overrides

```yaml
- uses: elioseverojunior/rust-toolchain@v1
  with:
    toolchain: nightly
    targets: wasm32-unknown-unknown, aarch64-apple-darwin
    components: clippy, rustfmt
    profile: minimal
```

Components can be specified as comma-separated, space-separated, or newline-separated:

```yaml

  with:
    components: |

      llvm-tools
```

Or inline:

```yaml
- uses: elioseverojunior/rust-toolchain@v1
  with:
    components: "clippy rustfmt llvm-tools"
```

### Profile Input

Set the Rust profile via input (overrides `rust-toolchain.toml`):

```yaml
- uses: elioseverojunior/rust-toolchain@v1
  with:
    profile: minimal
```

Valid profiles: `minimal`, `default`, `complete`

## Programmatic Usage

There is no barrel export — import the modules directly.

```ts
import { ToolchainSpecBuilder } from "./src/builder";
import { resolveChannel } from "./src/core";

// Fluent builder
const spec = new ToolchainSpecBuilder()
  .withChannel("stable")
  .withTargets("wasm32-unknown-unknown")
  .withComponents("clippy")
  .withProfile("minimal")
  .build();

// Targets and components are applied as separate commands, because
// --target/--component on `rustup toolchain install` are ignored for an
// already-installed toolchain.
spec.toRustupInstallCommand();
// => "rustup toolchain install stable --profile minimal"

spec.toRustupTargetAddCommands();
// => ["rustup target add --toolchain stable wasm32-unknown-unknown"]

spec.toRustupComponentAddCommands();
// => ["rustup component add --toolchain stable clippy"]

// Channel resolution — expressive forms resolve to a <major.minor> series,
// which rustup pins to the newest patch in that series.
resolveChannel("stable 6 months ago"); // => "1.93"
resolveChannel("stable minus 3 releases"); // => "1.94"
resolveChannel("1.62"); // => "1.62" (rustup picks the newest 1.62.x)
resolveChannel("nightly-2025-01-01"); // => "nightly-2025-01-01" (unchanged)
```

`--toolchain` is pinned explicitly on every `target add` / `component add`.
Without it rustup resolves through its override chain, so a `rust-toolchain.toml`
in the working directory would attach targets and components to _that_ toolchain
instead of the one just installed.

## `rust-toolchain.toml` Example

```toml
[toolchain]
channel = "stable"
targets = ["wasm32-unknown-unknown", "aarch64-apple-darwin"]
components = ["clippy", "rustfmt"]
profile = "minimal"
```

Values here are merged with the action inputs, and inputs win on conflict.

rustup also allows `path` instead of `channel` to select a local custom
toolchain. There is nothing for rustup to install in that case, so the action
fails with an explicit error rather than silently falling back to `stable` —
set the `toolchain` input if you need a channel.

## Toolchain Resolution

After installing, the action sets `RUSTUP_TOOLCHAIN` for the rest of the job.
rustup picks a toolchain in this order, highest first:

1. `+toolchain` shorthand (e.g. `cargo +beta build`)
2. **`RUSTUP_TOOLCHAIN`** ← set by this action
3. directory override (`rustup override`)
4. `rust-toolchain.toml`
5. the default toolchain

Because the action's resolved channel lands at position 2, "inputs override the
toml" holds when your later steps actually run `cargo` and `rustc` — not only at
install time.
